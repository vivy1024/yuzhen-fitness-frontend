## 智能健身动作监控系统

### 项目背景

大三时（2023年），我和舍友张金豪合作开发了一个智能健身动作监控系统。这是我第一次尝试将AI技术应用到健身领域。

### 技术方案

- **硬件**: 摄像头
- **算法**: MediaPipe姿态估计
- **功能**: 实时监控健身动作，识别错误姿势，提供纠正建议

### 老师的误解

项目展示时，老师质疑："这不就是调用API吗？有什么技术含量？"

这让我们很受挫。但现在回想起来，老师的质疑反而让我明白：**真正的价值不在于从零造轮子，而在于如何巧妙地组合现有工具，解决实际问题**。

这个理念，后来成为了玉珍健身开发的核心思想。

## 毕设：人脸微表情识别系统

### 项目概述

**题目**: 基于深度学习的人脸微表情识别系统  
**时间**: 2024年9月 - 2025年6月  
**GitHub**: [开源地址](https://github.com/vivy1024/facial-micro-expression-recognition)

### 技术架构

- **模型**: CNN + LSTM混合架构
- **数据集**: CASME II（中科院心理所微表情数据库）
- **识别率**: 76.2%（7种微表情分类）
- **框架**: PyTorch + OpenCV

### AI辅助开发的起点

毕设开发过程中，我开始大量使用AI辅助工具（Claude、ChatGPT）：
- 理解深度学习概念
- 调试PyTorch代码
- 优化模型架构
- 撰写论文

**关键转折点**: 我发现，AI不是替代我思考，而是帮助我更快地学习和实践。这让我意识到，**零基础也能通过AI辅助开发复杂系统**。

## 开发历程：从想法到三端系统

### 技术栈选择

**前端**: Vue 3 + Vite + PWA（移动端优先）  
**后端**: Laravel 11 + MySQL + Redis  
**AI服务**: Python + FastAPI + DAML-RAG框架  
**数据库**: Neo4j（知识图谱）+ Qdrant（向量库）  
**部署**: Docker + Docker Compose

**为什么选择这些技术**:
- Vue 3: 渐进式框架，学习曲线友好
- Laravel: PHP生态成熟，文档完善
- Python: AI生态最丰富，库最全
- Neo4j: 图数据库最适合健身知识建模
- Docker: 一键部署，环境一致

### 关键里程碑

**说明**: 2025年9月-11月是探索学习阶段，没有使用Git版本控制。真正的规范化开发从2025年12月11日开始，引入Git和Spec工作流，**不到1个月时间完成三端系统开发**！

## AI辅助开发过程分析

### 我犯过的错误

#### 错误1：过度依赖AI，不理解原理

**表现**: 
- 直接复制AI生成的代码，不理解为什么这样写
- 遇到问题时无法独立调试
- 代码风格不统一，到处都是AI的痕迹

**教训**:
- AI是助手，不是替代品
- 必须理解每一行代码的作用
- 学会调试和排查问题

**改进**:
- 要求AI解释代码逻辑
- 自己动手修改和优化
- 建立自己的代码规范

#### 错误2：架构设计混乱，频繁重构

**表现**:
- 早期没有整体架构规划
- 功能堆砌，代码耦合严重
- 多次大规模重构（v1.0 → v2.0 → v3.0 → v5.0）

**教训**:
- 架构设计要先行
- 模块化和解耦很重要
- 重构成本很高

**改进**:
- 学习设计模式和架构原则
- 参考成熟项目的架构
- 小步迭代，避免大规模重构

#### 错误3：忽视测试，导致Bug频发

**表现**:
- 早期没有任何测试
- 修复一个Bug，引入三个新Bug
- 上线后频繁回滚

**教训**:
- 测试是必须的，不是可选的
- 单元测试、集成测试、端到端测试都要有
- 测试驱动开发（TDD）很有价值

**改进**:
- 引入Vitest测试框架
- 编写160+个单元测试用例
- 每次提交前运行测试

#### 错误4：文档缺失，自己都看不懂

**表现**:
- 代码没有注释
- 没有API文档
- 过一段时间自己都不知道为什么这样写

**教训**:
- 文档和代码同样重要
- 好的文档能节省大量时间
- 文档是给未来的自己看的

**改进**:
- 建立完整的文档体系
- 代码注释要清晰
- 定期更新文档

#### 错误5：盲目追求新技术，忽视稳定性

**表现**:
- 频繁更换技术栈
- 引入不成熟的库
- 为了用新技术而用新技术

**教训**:
- 技术选型要慎重
- 稳定性比新颖性更重要
- 不要为了炫技而牺牲用户体验

**改进**:
- 选择成熟稳定的技术栈
- 新技术先在小范围试验
- 关注社区活跃度和文档质量

## AI辅助开发的正确姿势

### 1. 明确需求，不要让AI猜

**错误做法**:
```
"帮我写一个健身应用"
```

**正确做法**:
```
"我需要一个健身应用，包含以下功能：
1. 用户注册登录（邮箱+密码）
2. 动作库（1790+个动作，支持筛选和搜索）
3. AI训练计划生成（基于用户档案）
4. 训练记录和进度追踪

技术栈：
- 前端：Vue 3 + Vite + PWA
- 后端：Laravel 11 + MySQL
- AI服务：Python + FastAPI

请先设计数据库结构，然后实现用户注册登录功能。"
```

### 2. 分步实现，不要一次性完成

**错误做法**:
```
"帮我实现整个DAML-RAG框架"
```

**正确做法**:
```
"第一步：实现向量检索（Qdrant）
第二步：实现图谱检索（Neo4j）
第三步：实现约束检索（规则引擎）
第四步：整合三层检索引擎
第五步：添加缓存和监控"
```

### 3. 理解原理，不要盲目复制

**错误做法**:
- 直接复制AI生成的代码
- 不理解为什么这样写
- 遇到问题无法调试

**正确做法**:
- 要求AI解释代码逻辑
- 自己动手修改和优化
- 学习相关概念和原理

### 4. 建立规范，保持一致性

**错误做法**:
- 每次让AI生成代码，风格都不一样
- 命名混乱，结构不统一
- 代码质量参差不齐

**正确做法**:
- 建立代码规范（ESLint + Prettier）
- 统一命名规则和目录结构
- 定期Code Review

### 5. 善用工具，提高效率

**推荐工具**:
- **AI助手**: Claude、ChatGPT、GitHub Copilot
- **代码质量**: ESLint、Prettier、SonarQube
- **测试工具**: Vitest、Playwright、Postman
- **文档工具**: VitePress、Swagger、Mermaid
- **版本控制**: Git、GitHub、GitLab
